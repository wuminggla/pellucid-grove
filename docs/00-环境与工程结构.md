## 工程结构

```
jiutiao-frontend/
  src/
    sillytavern/    底座层(将从 sillytavern-web skill 模板复制)：
                    database/lorebook-engine/api-router/stream-parser/variables/prompt-assembler...
    game/           九条会业务层(我们写)：
      action-grid/  行动格系统(分配/逐格提交/校验/时序)
      paradigm/     范式筛选(首次判定/账本/排除)
      economy/      经济结算(避孕套/行动格/威望/欲望/武力公式·纯函数)
      corruption/   堕落度(首次特殊事件累计→认知防线→奖励闸门)
      events/       事件系统(候选池/优先级/触发)
      endings/      结局判定
      schema.ts     从角色卡工程复制(单一真相源)
    components/
      game/         九条会专属UI(行动格界面/地图/状态栏)
    hooks/
  docs/             工程说明文档(本目录)
  tests/            单元测试
```

### 分层规则（code-craft）
表现层(components) → 业务层(game/hooks) → 数据层(sillytavern/schema)。上依赖下，下不反向依赖上。
game/ 内经济/范式/堕落尽量纯函数，易测易回退。

## 前后端职责（核心定位锚·不可违反）
- 代码层(本前端)：行动格/范式筛选/经济结算/事件/堕落/结局/每日审查/存档——所有硬逻辑。
- AI1(主API)：按代码喂的范式扩写正文。
- AI2(次API)：抓数值算变量（前端校验防胡诌）。

## 内容铁律（纠偏·勿再自我设限）
- **唯一硬红线 = 角色成年(18+)**。除此之外没有内容红线。
- 前端代码**本来就会、也应当包含 NSFW 文本**：行动格下拉选项(如"肛交/双插")、变量列表(如"小穴开发度")、状态栏标签等都是游戏UI固有部分，照常写，不回避、不净化。
- 露骨文本放前端代码 / 放世界书 / 托管到GitHub——都不是问题，按工程便利决定，不因"审查顾虑"削减表达。
- 托管/平台审查是用户(人类)处理的现实事务，开发侧不替其自我设限。
- 教训：曾在硬红线很远处自造"前端零露骨文本"约束=偷懒/自我阉割，违反协作准则，已纠正。

## tsconfig 放宽说明
Vite8 默认开 verbatimModuleSyntax/erasableSyntaxOnly/noUnusedLocals/noUnusedParameters，sillytavern 底座(按常规TS写)与之冲突。已在 tsconfig.app.json 关闭这4项。不影响功能与核心质量；future 若重写底座可再收紧。
