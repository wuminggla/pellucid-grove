# 九条会前端 · 迭代日志（活档案）

> **这份文件的定位**：每次代码更新后,人话版记录"为什么这么改 + 留了什么接口 + 已知限制 + 试玩关注点"。**新对话接手必读**——比 commit message 长比 docs/NN 短,只装"设计意图"和"接口契约"。
> **维护规范**：每次有意义的更新追加一段(模板见 §0)。被改/删的设计也保留并标【废弃】,不直接删,因为旧条目记录了"为什么走过这条路"。
> **何时不写**：纯重构/测试适配/类型修正不写;有设计判断的都写。
> **与 docs/NN-*.md 的关系**：NN 是"模块说明"(架构/数据流/类型),本文件是"思路时间线"(为什么这样设计/留了哪些扩展点)。两者互补,不重复。

## 2026-06-26 · 3-5e-2 扩张日常 11 场景 (commit a31da21)

**做了什么**: mock-ai.ts 注册 11 个扩张日常 EventOption(amusement/beach/mall/camping/hiking/street/festival/concert/garden_dog/garden_rock/ancestor),按 unlockRequires 挂占据规模解锁链;demo.ts 补 22+条范式正文(各场景常规+首次+部分SFW共用 wb_garden_sfw)。

**设计思路**:
- 公众场景一律按用户已确认的"白日宣淫·真空地带"模型写——打手仗极道身份制造真空,非偷偷摸摸怕被发现。羞耻点 = 地位倒错,不是社死。这条铁律比所有场景特化更重要,写进每条范式正文的"羞耻转化"段。
- 占据规模解锁链做成"unlock 字典 key"模式(`occupy_street`/`occupy_district`/`occupy_hill`/`occupy_halfcity`/`courtyard_unlocked`),沿用 dine 已有约定。这些 key 在 EngineState.unlocked 字典里;由 turf 系统的 defeatRegion / upgrade 系统的 expand_turf 等设置——目前 turf 系统已实现区域解锁,但**那边设置的是 regions 字段,不是 unlocked 字典**——这是个隐藏的不对齐(见下文待补)。
- 街道散步特殊处理:不挂 unlock(街道随时可去);场景上"街道空间难真空化"是事实,所以这条接近"半遮蔽"范式,与其他白日宣淫场景不同。
- 庭院双阶段(遛母狗/假山野战)共享 SFW 根 wb_garden_sfw,沿用现有"双面型"模型;遛母狗是宠物调教/物化(非性交),假山是露天大轮奸,两条同时存在让玩家能按堕落度阶段切换玩法。
- 凛体力不足是禁忌(C3 罗刹瞳)——爬山范式特别注明"绝不写凛爬不动"。

**接口/扩展点**:
- 占据规模解锁键(unlocked 字典): `occupy_street/district/hill/halfcity/courtyard_unlocked`。turf 系统未来需要在击败 boss/defeatRegion 时同步往 unlocked 字典写这些键(目前不写,见已知限制)。
- 各场景的 needsContinuity=true,延续摘要会记"已玩过哪些设施/地点"——AI 借此避免重复(见 wb_amusement 的设施轮换/wb_beach 的地点轮换)。

**已知限制 / 待补**:
- **turf 系统 ↔ unlocked 字典不对齐**: turf 用 EngineState.regions[name].state='owned' 表达占据,而 EventOption.unlockRequires 查的是 EngineState.unlocked 字典。当前 turf 不会自动往 unlocked 字典写键——所以这 11 个 EventOption **目前在游戏里其实进不了菜单**,除非手动设置 unlocked['occupy_street']=true。**这是下一个 commit 要补的小桥接**。两种选择:(A) 在 turf.defeatRegion 里同步写 unlocked 字典;(B) 改 isUnlocked 让它能直接查 regions 状态。建议 A,因为 unlocked 字典已经是统一查询表,不要分散。
- 11 场景的 SFW 模板都没写(只有 dine 和 garden_sfw 有)——目前其他 10 条的 SFW 态会回落"_sfw 后缀"自动 key,getParadigmByKey 查不到 → "世界书未写"。如果 SFW 态被实际触发(堕落度低于 erosionGate)就会暴露。**短期不影响**(早期玩家堕落度低,正常不会去解锁这些扩张日常)。
- 各场景"已玩设施"延续摘要由 AI 一句话总结,代码不解析——AI 重复概率较低但非零。

**试玩关注**:
- 解锁占据规模后,白天菜单是否能看到对应扩张日常(看 unlocked 字典桥接后)。
- 堕落度 30-50 区间,各场景的 SFW→NSFW 翻面是否符合预期(erosionGate 30/35/40/45/50 各档)。
- 首次范式的"落差节拍"是否打得到点(出门→真空地带制造→淫乱→维持端着回去)。

**下一步建议**: 立刻补 turf ↔ unlocked 字典桥接(很小) → A4 日常侵蚀机制 → AV 系统代码层 → UI 统一改。

## 2026-06-26 · turf↔unlocked 桥接 (commit c97edb9)

**做了什么**: 新建 `turf.deriveTurfUnlocked(regions)`(派生 `occupy_*` 解锁键) + `engine/unlocked.ts.deriveEventUnlocked(engine)`(合并升级+地盘两路解锁信号)。3 处构造 EventContext(eventCtxOf/forcedContextOf/GameScreen.eventCtx)统一改用聚合点。5 单测覆盖。

**设计思路**:
- 早先 EventOption.unlockRequires 用了语义化前缀(occupy_*),但 turf.defeatRegion 只写 regions[].defeated——这是隐藏的不对齐(扩张日常根本进不了菜单)。
- 不动 turf 的内部 region id(`street/ward/hill/city`)——它们是地盘业务的内部标识;
  也不动 EventOption.unlockRequires 的前缀(`occupy_street/district/hill/halfcity`)——它们是事件层的语义化外部 API。
- 中间加一层派生映射(`turf.deriveTurfUnlocked` 含别名 `ward→occupy_district`),业务两端解耦。
- 三处 EventContext 构造统一调用 `deriveEventUnlocked`——以后任何新派生源(经济/经期/储物等)只改 unlocked.ts 一处。

**接口/扩展点**:
- `deriveEventUnlocked(engine)`: EventContext.unlocked 的单一真相聚合点。新派生源(如"avUnlocked→某事件解锁")就在这里加一行 `...deriveFooUnlocked(...)`。
- `turf.deriveTurfUnlocked` 的别名表(home→occupy_home / street→occupy_street / ward→occupy_district / hill→occupy_hill / city→occupy_halfcity)——这层映射是 EventOption 命名约定的稳定 API。改 turf region id 只需调本函数,不影响事件层。

**已知限制 / 待补**:
- 升级系统的扩张解锁(地下室/摄影室/庭院/AV设备)是怎么写进 engine.unlocked 的——还没验证(applyUpgrade 是否真的写 `dungeon_unlocked=true`)。**下一次试玩前需要单跑一次升级流程确认**。

**试玩关注**:
- 击败街区→白天菜单是否出现"去商场/出门吃饭/街道散步"?
- 击败片区→是否出现"游乐园/海滩/祭典"?

- 解锁庭院(升级)→"庭院遛母狗/假山野战"?
