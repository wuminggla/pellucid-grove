11. 文档体系: 工作区根 README + v2 README + v1 归档 README + 迁移记录-v1到v2.md

**设计思路**:
- v1 用 iframe + 自创 postMessage 协议跟酒馆生态不通。试跑暴露 iframe 被酒馆 sanitize 剥离,**致命**(详见 `迁移记录-v1到v2.md`)。
- v2 用 tavern_helper_template 的酒馆生态主流方案:Vue + Webpack 5 输出单文件 HTML + jQuery.load 同源 AJAX 注入。
- jQuery `$('#mount').load(URL)` 同源拿 HTML 文本,**脚本直接操作 DOM 绕过 sanitize**;testingcf.jsdelivr.net 镜像对 .html 返回正确 MIME。
- 业务代码(`game/` 16 模块·纯 TS·262 单测)零改动复用——这是把"两版工程"做成"两层皮"的关键投资,以后再换工程外壳还能复用。
- 部署机制大幅简化: 不再需要 GitHub Pages / gh-pages 分支 / GitHub Actions —— **直接 `git push` 把 dist 推到 master 分支**,jsdelivr 自动同步。
- 锁版本用 commit sha(`@<sha7>`)而非 tag(tag 在 master 但 dist 也在 master,没冲突)。

**接口/扩展点**:
- **业务代码**: 16 个 game 模块对 UI 框架零依赖,可继续用纯函数 + 单测开发。下批 UI 翻译时直接 import。
- **MVU 双向绑定**: 模板内置的 `@util/mvu` 提供 `defineMvuDataStore(Schema, ...)` —— Pinia store 直接绑定 MVU 变量,**免费解决 v1 担心的状态同步问题**。下批 UI 翻译时落地。
- **入口扫描**: webpack 自动扫 `src/**/index.{ts,tsx,js,jsx}` 当 entry,每个 entry 产出独立的单文件 HTML。未来要加新界面(比如设置面板)只需建新 `src/<name>/index.ts` + `index.html` + `App.vue`,自动出 entry。
- **多镜像 fallback**: 状态栏正则 HTML 已内置 testingcf → cdn.jsdelivr.net 回落逻辑,国内网络可靠性提升。

**已知限制 / 待补**:
